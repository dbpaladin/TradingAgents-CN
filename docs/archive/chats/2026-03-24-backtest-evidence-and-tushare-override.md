# 会话归档：回测资金证据排查、性能瓶颈修复与第三方 Tushare 覆盖

- 日期：2026-03-24
- 主题：排查“资金证据不完整”的真实原因；修复回测超时与强制买入问题；确保第三方 Tushare endpoint 在系统中实际生效
- 执行方式：日志排障 + 代码修复 + 前端显示调整 + 测试验证 + 文档归档

## 用户诉求

用户围绕最新回测连续提出以下需求：

1. 详细分析“资金证据不完整”等影响交易策略的问题，并给出修复方案。
2. 排查整体回撤/性能的主要瓶颈。
3. 调整回测明细中的耗时单位展示。
4. 去掉没有交易机会时的“强制买入”保护。
5. 明确“资金证据不完整”是否来自数据源问题。
6. 确认 Tushare 使用的是第三方 endpoint + token，而不是官方地址，并确保代码真正按此执行。
7. 最后归档聊天记录、更新相关文档并提交 GitHub。

## 关键结论

### 1) 最新回测 `0交易 / 0回撤` 的主因不是策略优秀，而是分析超时

日志显示回测多次在 AI 分析阶段触发长超时，随后整段被降级为 `HOLD`，这才产生了“零交易、零回撤”的表象。

### 2) “资金证据不完整”是证据覆盖不足，不等于程序异常

它通常来源于：

- `moneyflow`
- `margin_detail`
- `hk_hold`

等资金面子证据没有同时取全。原因可能是数据源未返回、股票未命中、当天无披露、或该口径本身不适用。

### 3) 真正会放大问题的是下游决策链路

原先分析师、交易员和风控提示词会把“证据不完整”放大成偏保守甚至偏空判断，因此本轮修复重点之一是把“缺数据”重新界定为“降置信度”，而不是“直接利空”。

### 4) 第三方 Tushare 之前只在 provider 层生效，manager 层还可能被数据库旧配置拦住

数据库里的 `tushare.enabled=false` 和官方 endpoint 占位配置，会让 `DataSourceManager` 提前跳过 Tushare。后来已补齐该规则，使其与 provider 层一致。

## 实施改动

### A. 资金面证据完整性修复

- 新增证据完整度分数、标签、缺失项、数据质量说明
- 将北向/融资融券缺失明确标为“数据缺口”
- 调整提示词，禁止把缺口直接解释为卖出依据

### B. 回测稳定性与真实性修复

- 增加回测专用运行时收敛，减少超时风险
- 长区间任务主动切轻量模式
- 移除“无买点时强制买入”的入场保护
- 前端耗时展示改为 `ms / s / 分钟 / 小时` 自适应

### C. 第三方 Tushare 覆盖补齐

- 在数据源管理层增加 `.env` 第三方 endpoint 检测
- 当 `TUSHARE_ENDPOINT` 为自定义地址且 token 有效时，按 `.env` 启用 Tushare
- 数据源优先级顺序同步修复，避免回测降级链路漏掉 Tushare
- 新增回归测试保护该行为

## 结果

1. 回测结果不再被强制入场保护污染。
2. “资金证据不完整”不再被误导为天然偏空。
3. 第三方 Tushare 接入在 provider 和 manager 两层规则已统一。
4. 本轮修复说明、会话归档和更新日志已同步补齐。
