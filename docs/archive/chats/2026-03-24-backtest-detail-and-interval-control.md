# 会话归档：回测0交易修复、明细可观测增强与周期控制回归用户

- 日期：2026-03-24
- 主题：修复回测“0交易”死局；增强回测明细中的模型与耗时展示；取消回测周期自动改写
- 执行方式：在线监控任务日志 + 后端修复 + 前端展示增强 + 测试验证 + 文档同步

## 背景

用户连续反馈回测“跑完但 0 交易”，并在日志中看到：

- `AI买入=0`
- `AI卖出>0 但执行失败`
- `AI持有/降级 HOLD`

同时用户明确提出两点体验诉求：

1. 回测结果明细需要看到“用的什么模型”和“每步推理耗时”。
2. 回测速度模式不应被系统自动改写为 3 天或 5 天，周期应由用户自己判断。

## 关键诊断结论

### 1) 回测 0 交易并非总是超时导致

通过任务状态与日志联查确认，部分任务是正常完成的（`status=completed`），但出现了“空仓收到 SELL 信号”的不可执行路径，导致：

- 信号层面有动作（SELL/HOLD）
- 执行层面仍是 0 成交
- 最终触发 0 交易诊断

### 2) 用户感知到的“报错”有一部分来自行情定时采集而非回测引擎

日志显示定时任务存在：

- Tushare `rt_k` 无权限
- AKShare 临时连接中断

这会制造错误噪音，但不等于回测任务失败。

## 实施改动

### A. 回测执行逻辑修复（避免 0 交易死局）

- 增加“执行前信号归一化”
  - 空仓 `SELL` -> `HOLD`
  - 持仓 `BUY` -> `HOLD`
- 增加“连续空仓无买入”入场保护，避免整段 0 交易
- 相关行为写入诊断统计，方便追踪触发次数

### B. 回测明细增强（模型与耗时可见）

- 为交易明细增加字段：
  - `ai_provider`
  - `ai_model`
  - `ai_runtime_mode`
  - `analysis_elapsed_ms`
  - `decision_elapsed_ms`
  - `execution_elapsed_ms`
  - `day_elapsed_ms`
- 前端回测结果页新增“AI推理明细（按天）”表格，展示模型信息与分步耗时

### C. 回测周期控制策略调整（尊重用户）

- 关闭按负载自动改写 `decision_interval_days` 的逻辑
- 不再自动从 1 改为 3/5
- 回测周期完全按用户输入执行

### D. 行情采集降噪与回退增强

- 已确认 `rt_k` 无权限时跳过重复调用，避免重复 ERROR 噪音
- 单轮采集在失败时继续尝试备用源（Tushare -> AKShare 东财 -> AKShare 新浪）
- 对可预期临时网络错误降级日志级别

## 验证

- `tests/test_backtest_service.py` 全通过（新增用例覆盖）
- `tests/services/test_quotes_ingestion_and_enrichment.py` 通过（含新增跳过无权限场景用例）
- 回测任务在线监控显示：任务可完成，且可区分“回测完成”与“行情采集噪音错误”

## 用户可见结果

1. 回测明细中可直接看到模型来源和分步耗时。
2. 回测周期不再被系统自动改写，用户设置即执行策略。
3. 日志中与实时行情采集相关的已知无权限场景噪音显著降低。
