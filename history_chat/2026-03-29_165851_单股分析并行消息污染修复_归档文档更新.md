## 问题现象

- 单股分析任务在运行到多分析师阶段后失败。
- 服务层日志表面错误为：
  - `TypeError: 'NoneType' object is not iterable`
  - 更早一轮并行修复中还出现过 `Attempting to delete a message with an ID that doesn't exist`
- 进一步展开真实堆栈后，核心异常为：
  - `openai.BadRequestError: No tool output found for function call ...`

## 根因定位

本轮排查结合了仓库近期提交、线上错误日志和本地同步复现，最终确认根因不是行情数据本身，而是 2026-03-29 引入的“分析师并行化”对正式单股分析链路造成了副作用：

1. 多个分析师分支共享同一份 `messages` 状态。
2. 为了绕过并行 `RemoveMessage` 冲突，`create_msg_delete()` 被改成直接返回 `{}`。
3. 这样虽然避开了 `RemoveMessage` 崩溃，但各分析师的 tool call / tool output 历史被继续保留并互相污染。
4. 某个分析师第二轮调用 LLM 时，消息中混入了其他分支的 function call 记录，却没有对应 tool output，于是 OpenAI 兼容接口返回：
   - `No tool output found for function call ...`

## 实施修复

### 1) 恢复正式单股分析为串行分析师执行

文件：`tradingagents/graph/setup.py`

- 将 `setup_graph()` 从“所有分析师从 `START` 并行扇出”恢复为“按选中顺序串行串接”
- 保持正式分析链路的消息隔离和工具调用闭环
- 回测专用轻量图 `setup_backtest_graph()` 继续保留并行结构

### 2) 为消息清理增加双模式

文件：`tradingagents/agents/utils/agent_utils.py`

- `create_msg_delete()` 新增 `parallel_safe` 参数
- 默认模式：
  - 删除当前分支消息
  - 添加 `HumanMessage("Continue")` 占位
  - 用于正式单股分析串行图
- 并行安全模式：
  - 返回 `{}`，不执行删除
  - 仅用于回测轻量并行图，避免 `RemoveMessage` 冲突

### 3) 补充回归测试

文件：`tests/test_msg_delete_behavior.py`

- 验证默认模式会正常生成 `RemoveMessage + Continue`
- 验证并行安全模式返回 `{}`，避免并行删除冲突

## 备注

本次修复的最终结论是：

- 回测轻量图可以继续保留并行分析师执行
